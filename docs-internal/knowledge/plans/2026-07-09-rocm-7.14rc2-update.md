# ROCm 7.14.0rc2 update: bump version refs + sync amdsmi assets

- **Date:** 2026-07-09
- **Author:** praveen
- **Branch:** `feature/rocm-update-skill-refresh`
- **Related PR(s):** TBD
- **Related issue(s) / JIRA:** GPUOP-970

## Context

Move the Device Metrics Exporter from ROCm 7.14.0 RC1 to **RC2**. RC1 was
synced in #1440 (`71fc15292`); the RC1 URL bump was #1432 (`5da02f838`). RC2
follows the same tarball-driven path — no gpu-agent repo edit, no host-side
gpuagent build (gpuagent compiles inside the exporter docker build as of the
2026-06-16 source-build pivot).

RC2 tarball:
`https://rocm.prereleases.amd.com/tarball-multi-arch/therock-dist-linux-multiarch-7.14.0rc2.tar.gz`
(HTTP 200, ~8.5 GB, last-modified 2026-07-08).

This update also lands a refresh of the `/rocm-update` skill
(`.claude/skills/rocm-update/SKILL.md`), which still documented the retired
submodule + host-side gpuagent workflow. Rewritten to match the current
tarball-driven, in-docker-build workflow.

## Approach

1. Bump `ROCM_VERSION` → `7.14.0rc2` and `ROCM_TARBALL_URL` → the RC2 tarball in
   both `Makefile` and `dev.env`; update the `dev.env` provenance comment.
2. `make amdsmi-sync-assets AMDSMI_TARBALL_FORCE=1` — refresh
   `assets/amd_smi_lib/x86_64/RHEL9/lib/{libamd_smi.so*, amdsmi.h,
   librocm_sysdeps_*.so*}` and `version.txt` from the RC2 tarball. UBUNTU22/24
   are symlinks to RHEL9 and update automatically.
3. `make docker` (default `AMDSMI_FROM_TARBALL=0`) against the synced assets;
   gpuagent clones at `GPUAGENT_COMMIT` and links in-image against the same
   amdsmi.
4. Smoke test `gpu_*` / `amd_gpu_*` metrics.

## Scope

- **In scope:** `Makefile` + `dev.env` version bump, RC2 amdsmi asset sync
  (RHEL9 source of truth), `/rocm-update` skill refresh, and the amdsmi
  source-build pin — `AMDSMI_BRANCH` → `release/therock-7.14`, `AMDSMI_COMMIT`
  → `53a7a4f3fe6019a551506285f9f2bb86dfddf9b4` (consumed by the

- **Out of scope:** `assets/version.yaml` `7.13` references — stale metadata

## Key finding: amdsmi binaries unchanged from RC1

Running `make amdsmi-sync-assets AMDSMI_TARBALL_FORCE=1` against the RC2 tarball
produced **byte-identical** amdsmi artifacts to RC1: `libamd_smi.so.26.5.0`
(5,803,976 bytes, unchanged), `amdsmi.h`, and all 5 tracked
`librocm_sysdeps_*.so*` are identical. The only tracked change from the asset
sync is `version.txt` (`therock-7.14.0rc1` → `therock-7.14.0rc2`).

Net effect: the RC1→RC2 amdsmi component is a no-op at the binary level. The
functional change RC2 delivers lives in the ROCm runtime tarball consumed at
docker build time (`install-rocm-tarball.sh` via `ROCM_TARBALL_URL`), not in the
committed amdsmi assets.

Verified by SHA256 (not just file size) against the RC1 sync commit `71fc15292`
— all 7 assets identical:

| Asset | RC1 vs RC2 |
|---|---|
| `libamd_smi.so.26.5.0` | identical |
| `amdsmi.h` | identical |
| `librocm_sysdeps_drm.so.2` | identical |
| `librocm_sysdeps_drm_amdgpu.so.1` | identical |
| `librocm_sysdeps_mnl.so.0` | identical |
| `librocm_sysdeps_nl_3.so.200` | identical |
| `librocm_sysdeps_nl_genl_3.so.200` | identical |

## Real-hardware validation: justified skip for the amdsmi path

This dev box is a VMware VM (no AMD GPU), so mock-only verification ran here.
Real-HW validation is **deliberately skipped** for the `gpu_*` (amdsmi) metric
path, on a two-part identity argument:

1. **RC2 amdsmi ≡ RC1 amdsmi** (SHA256 table above). The `.so` shipped at
   runtime and the `.so` gpuagent links against are byte-for-byte the RC1
   binaries.
2. **RC1 amdsmi was validated on real hardware** — see the RC1 sync plan
   (`2026-07-07-rocm-7.14rc1-amdsmi-sync.md`): single-GPU host (AMD Radeon
   Graphics), 88+ live metrics, `rocpctl` resolving all 28 shared libs, and
   `amd-smi` vs `/metrics` matching on power/clocks/VRAM.

Byte-identical binaries cannot produce different runtime metric behavior, so the
RC1 hardware run transitively covers RC2's amdsmi path. This satisfies the
GPUOP-970 acceptance criterion (basic `amd_gpu_*` metrics).

**Caveat — not covered by this argument:** the RC2 *runtime tarball*
(`ROCM_TARBALL_URL`) is only installed when that build-arg is explicitly passed
(CICD/hourly `docker-cicd`, profiler/`GPU_PROF_*` libs). The default `make
docker` used here does not consume it. Any RC1→RC2 delta lives there and would
need a hardware profiler-metrics check in a CICD-built image — out of scope for
this JIRA's acceptance criteria.

- `make amdsmi-sync-assets AMDSMI_TARBALL_FORCE=1`: `version.txt` →
  `therock-7.14.0rc2`; `.so`/header/sysdeps byte-identical to RC1 (see finding

  `GPUAGENT_COMMIT=9af0bf5b` and compiled in-image; RC2 amdsmi + 5 sysdeps

- Real-hardware validation of the `gpu_*` amdsmi path is **skipped by
  identity argument** (see section above) — RC2 amdsmi is SHA256-identical to
  RC1, which was hardware-validated. Mock here only sanity-checks build/wiring.

## Build gotchas hit (host-side, non-blocking)

- `make docker-compile` / `build-exporter.sh` uses `docker run -it` → fails
  under a non-TTY harness ("the input device is not a TTY"). Set `CI=1` to drop
  `-it` (`DOCKER_IT_FLAGS := $(if $(CI),,-it)`).
- With `-it` gone, the build container's login shell (`su - praveen`) loses the
  Go PATH — Ubuntu `.bashrc` early-returns for non-interactive shells, so
  `gopkglist` fails with `go: No such file or directory`. Fix: prepend
  `export PATH=/usr/local/go/bin:$HOME/go/bin:$PATH` to the in-container command.
- `.claude/bin/build-docker.sh` runs under `set -u` and sources `dev.env` (which
  uses Makefile `?=` syntax bash can't parse) → `DOCKER_REGISTRY: unbound
  variable`. Export `DOCKER_REGISTRY` before invoking.

## Risks and rollback

- Low risk — tarball URL format unchanged, only the version suffix bumped, same
  build path as RC1. Rollback: revert the Makefile/dev.env bump and restore
  `assets/amd_smi_lib/` from git history
  (`git checkout HEAD~1 -- assets/amd_smi_lib/`).
- If the RC2 `.so` version differs from `26.5.0`, no Dockerfile edit is needed —
  both builder and runtime stages derive the major/symlink names dynamically.
