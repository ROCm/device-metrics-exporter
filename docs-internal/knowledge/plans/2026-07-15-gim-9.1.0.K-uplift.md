# Uplift GIM SMI lib to 9.1.0.K + refresh SR-IOV walker asset

- **Date:** 2026-07-15
- **Author:** Bhanu Kiran Atturu
- **Related PR(s):** chore/gim-9.1.0.K
- **Related issue(s) / JIRA:** GPUOP-880 (follow-on)

## Context

AMD released GIM 9.1.0.K
(https://github.com/amd/MxGPU-Virtualization/releases/tag/9.1.0.K). The exporter
was pinned at GIM 8.7.0.K in the build refs and shipped a 9.0.0.K
`libgim_amd_smi.so` in `assets/gim_smi_lib/`. This bumps the GIM host-side SMI
library to 9.1.0.K and refreshes the prebuilt SR-IOV `gpuagent` walker that
consumes it. (The sw repo is intentionally out of scope for this change.)

9.1.0.K expands `amdsmi_metric_name_t` from 73 to 171 entries — purely additive
(new UBB/VR/SVI temps, FCLK/LCLK, shutdown/throttle temps, `SYSTEM_POWER_UBB_POWER`,
and mid-enum `THROTTLE_PROCHOT/PPT_ACTIVE` already handled in the 9.0 walker). The
GIM walker is positional and compiled against the vendored header, so a
re-vendor + recompile picks up the new numbering safely and unhandled entries fall
through to the existing sentinel default.

## Approach

- Bump the `libgimsmi` submodule `8.7.0.K` → `9.1.0.K` (tag `05008cb`),
  `.gitmodules` branch → `release/9.1.x.K-rc`, and `Makefile`
  `GIMSMI_BRANCH`/`GIMSMI_COMMIT` → `release/9.1.x.K-rc` / `9.1.0.K`.
- Re-stage `assets/gim_smi_lib/x86_64/RHEL9/lib/{libgim_amd_smi.so,amdsmi.h}` +
  `version.txt` to 9.1.0.K. RHEL9-built (GLIBC floor 2.34, SONAME `libamdsmi.so`);
  UBUNTU22/24 lib dirs are symlinks to RHEL9 and update automatically.
- Fix the GIM build entrypoint (`tools/gimlib-builderimage/entrypoint.sh`):
  9.1.0.K smi-lib enables `AMD_SMI_NIC_SUPPORT` by default and its main `.so`
  link now needs `nic/build/libamdsminic.a` + `nic/build/nl/build/libnl.a`. The
  entrypoint previously only ran `make` in `smi-lib`, so the link failed on the
  missing `libnl.a`. Build the nic target first (`make -C nic`) before `make`.
- Refresh the prebuilt SR-IOV asset `assets/gpuagent_sriov_static.bin.gz` with a
  `gpuagent_gim` built from gpu-agent `feat/gim-9.1.0.K` (re-vendored 9.1.0.K lib
  + UBB node power wiring in the GIM path). This is the only place the GIM walker
  ships in DME — the standard image builds gpuagent from source at
  `GPUAGENT_COMMIT`, which has no GIM walker.

### Alternatives considered

- Wiring all ~96 new 9.1.0.K metrics (UBB/VR/SVI temps) end-to-end — deferred.
  Most are MI350X-specific and unvalidatable on the MI300X test host; the non-GIM
  amdsmi path doesn't wire them either. Only UBB node power was wired (mirroring
  the non-GIM path), the rest fall through to sentinel.

## Scope

- **In scope:** GIM submodule + build-ref bump, `gim_smi_lib` asset refresh, gimlib
  build-order fix, SR-IOV `gpuagent` asset refresh.
- **Out of scope:** the gpuagent code change itself (gpu-agent `feat/gim-9.1.0.K`,
  separate repo/PR); `gpuctl.gobin` (GIM-independent CLI, unchanged); surfacing
  `amd_gpu_ubb_power` as an exporter metric (needs a DME proto/mapping change +
  MI350X HW); the sw repo.

## Validation

- **Local build:** `libgim_amd_smi.so` built in the RHEL9 gimsmi builder
  (`make -C nic` then `make`) → SONAME `libamdsmi.so`, GLIBC floor 2.34, exports
  `amdsmi_get_npm_info` + `amdsmi_get_node_handle`. `gpuagent_gim` built clean in
  the RHEL9 gpuagent builder.
- **Image:** SR-IOV image rebuilt via the full `make docker-sriov` asset path
  (Dockerfile.sriov consuming the committed `gpuagent_sriov_static.bin.gz`).
- **Hardware (banff, 8× MI300X, GIM 9.1.0.K loaded via official DKMS deb):** e2e —
  exporter comes up `deployment_mode="hypervisor"`, **1929 `amd_` series / 124
  families** across 8 GPUs, all throttle/violation families decode (positional
  walker correct under the 171-entry enum), core metrics sane (power 139 W, VRAM
  196592 MB, junction 38 °C), **zero sentinel leaks**, gpuagent stable — no
  regression vs 9.0.0.K.

## Risks and rollback

- **Known risks:** a GIM smi-lib bump can shift metric decoding; mitigated by the
  additive-only enum change and the banff MI300X e2e run through the real asset
  build path.
- **Rollback:** revert this commit to restore the 9.0.0.K `libgim_amd_smi.so`,
  the 8.7.0.K submodule pin, and the prior SR-IOV asset. No config/schema
  migration involved.
